# Verified Codebase Orientation

This codebase has a **formally verified core**. The file `src/domain.ts`
contains domain logic that is mathematically proven correct via LemmaScript
and Dafny. Everything outside of it is unverified shell.

## The boundary

```
VERIFIED (domain.ts)          UNVERIFIED (everything else)
─────────────────────         ──────────────────────────────
State interfaces              React components
Op union                      Hooks
applyOp / apply               Store implementations
Invariant (wellFormed)         Network / WebSocket
Aggregation (heatmap, etc.)   Database / storage
Query functions               Routing, auth, config
Relational lemmas             CSS, assets
```

## Rules

### Never duplicate domain logic outside domain.ts
If a component needs a count, a comparison, a filter, or any derivation from
state, it must call a function exported from `domain.ts`. Do not recompute
it in JSX, in a hook, or in a utility file.

### State changes only through the verified apply function
The store's `dispatch` calls `applyOp` (or `apply`). That is the only place
state mutates. Components create Op data and hand it to dispatch. No direct
state mutation anywhere.

### The hook is the only caller of verified read functions
One hook (e.g., `useQuorum`) calls `heatmap`, `isBest`, `whoIsFree`, etc.
Components receive the results as props or return values. Components never
import from `domain.ts` directly.

### New domain logic goes in domain.ts with annotations
If you need new domain behavior:
1. Add the function to `domain.ts` with `//@ verify` and appropriate
   `//@ ensures` / `//@ requires`
2. Run `lsc regen --backend=dafny src/domain.ts` then `dafny verify`
3. Iterate until proofs pass
4. Export and call from the hook

### Never edit domain.dfy.gen
This file is auto-generated by LemmaScript. Proof additions go in
`domain.dfy`. Use `lsc regen` (not `gen`) after TypeScript changes to
preserve existing proofs.

### Do not add //@ assume
`//@ assume` bypasses proof obligations. It is never acceptable in
production code. If a proof is stuck, add helper lemmas or weaken the
ensures to something provable.

## Common mistakes to avoid

- Adding a `filtered.length` or `array.reduce` in a component instead of
  calling a verified query from domain.ts
- Creating a second source of truth for derived state (e.g., a separate
  "vote count" variable maintained in the UI)
- Calling `applyOp` directly from a component instead of going through
  the store's `dispatch`
- Editing `domain.dfy.gen` instead of `domain.dfy`
- Using `lsc gen` instead of `lsc regen` (destroys proof additions)
